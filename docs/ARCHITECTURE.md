# Promptcase Studio 아키텍처

## 실행 흐름

```text
GUI 입력
  -> Change Collector (Git / 수정일 / 수동 목록)
  -> Deterministic Project Index
  -> Dynamic Context Builder
  -> Prompt Builder
  -> Gemini | Qwen | Mock Provider
  -> Structured Response Parser
  -> Coverage and Language Quality Review
  -> Excel Template Writer
  -> 사용자 지정 경로로 다운로드
```

## 변경 범위

- Git: 지정한 시작일 및 종료일 범위의 commit에서 A/M/D/R 상태를 읽는다. 종료일이 오늘이면 현재 working tree도 포함하고, 과거이면 현재 변경이 섞이지 않게 제외한다.
- 수정일: 현재 존재하는 파일 중 시작일부터 종료일까지 수정된 파일을 찾는다. 양 끝 날짜를 모두 포함한다.
- 수동 목록: 비Git 환경의 변경 파일과 삭제 파일을 보완한다.
- 사용자 변경 요약: feat와 refactor 같은 개발자 설명을 파일 경로와 분리해 프롬프트에 유지한다.
- 같은 파일이 여러 입력에서 발견되면 Git, 수동, 수정일 순으로 신뢰도를 병합한다.

기본 범위는 이번 달 1일부터 오늘까지다. 수정일 스캔만으로는 이미 삭제된 파일을 알 수 없으므로 삭제는 Git 또는 수동 입력이 필요하다.

## 컨텍스트 선택

1. 변경 파일마다 독립 예산을 보장하고 Git diff를 현재 소스보다 먼저 배치한다.
2. 큰 파일은 의뢰와 변경 요약의 핵심어, 조건문, 함수, SQL 주변을 우선 발췌하고 import만으로 예산이 소진되지 않게 한다.
3. Java, Kotlin, Python, TypeScript import와 MyBatis namespace, resultType, refid, SQL id, 프론트 API와 Spring endpoint를 참조 신호로 만든다.
4. Controller에서 Service와 Mapper 또는 DTO까지 정확한 참조를 한 단계 더 확장한다.
5. 역할 계층의 다양성과 의뢰 연관도를 함께 반영해 상위 관련 파일을 선택한다.
6. 파일 수가 많아도 모든 변경 파일의 근거 헤더를 유지하며 기본 420000자 프롬프트 예산을 설정에서 조정한다.
7. diff, 조건, 호출, SQL과 식별자 근거를 층화해 같은 키워드 반복이 예산을 독점하지 않게 한다.
8. UTF 계열 BOM과 XML 인코딩 선언 및 CP949 소스를 bounded read로 처리한다.
9. 제외 파일, 선택 이유, 프롬프트, 근거 전용 corpus, 초안과 검토본, 품질 보고서와 실행 로그를 `runs/<run-id>/`에 기록한다.

## 구조화 응답

AI는 프로그램 파일 목록을 결정하지 않는다. 프로그램 정보는 change manifest에서 만들고,
AI는 테스트케이스와 테스트 결과 문안만 JSON으로 반환한다. 응답은 문서 생성 전 필수 필드와
리스트 길이, 근거 없는 화면 식별자, 중복 문안과 금지 문자를 검증한다. Gemini와 Qwen은 같은
프롬프트와 같은 parser 계약을 사용한다. 근거 검증에는 시스템 지시와 JSON 스키마 예시를 제외한
개발 의뢰, 변경 목록과 실제 선택 소스만 사용한다.

형식 검증을 통과한 초안은 변경 앵커와 정상, 부정, 경계, 권한, 오류, 삭제, 회귀 시나리오의
커버리지를 다시 계산한다. 별도 품질 검토 호출은 구현 존재 여부를 사전조건으로 쓴 문장,
의미상 중복, 누락된 변경 분기와 어색한 한국어를 교정한다. 검토본이 품질 지표를 개선하지
못하면 기존 검증 초안을 유지한다. 기본 완료 정책은 계약을 통과한 최선본을 검토 필요 상태로
다운로드할 수 있게 하며, 선택 가능한 엄격 정책에서는 핵심 문장 문제가 남으면 다운로드를 차단한다.

사용자 의뢰에서 저장, 조회조건, 다운로드, 업로드, 삭제, 권한, 경계값, 오류, 잠금,
활성·비활성 또는 변경 없음 조건을 찾으면 조건별 의무 항목으로 관리한다. 각 항목은 테스트 절차의
실제 조작과 테스트내역·예상결과·결과화면의 관찰 가능한 판정이 함께 있어야 반영으로 인정한다.
`삭제: api/x.ts`처럼 파일 경로만 적힌 행은 필수 시나리오로 승격하지 않는다. 첫 검토에도 필수
항목이 남으면 설정된 횟수만큼 교정한다. 그래도 남으면 기본 정책에서는 품질 진단과 함께 최선본을
생성하고, 엄격 정책에서만 문서 생성을 차단한다.

## Provider 제한시간과 재시도

- Gemini와 Qwen의 기본 제한시간은 300초다.
- 한 요청은 최초 시도를 포함해 최대 3회까지만 시도한다.
- HTTP 응답 전 연결 실패와 408, 425, 429, 500, 502, 503, 504만 재시도한다.
- HTTP 응답 또는 SSE chunk 수신 후의 timeout은 중복 POST 위험 때문에 전송 계층에서 재시도하지 않는다.
- 완전한 응답을 받은 뒤 문서 계약 검증이 실패하면 오류와 이전 응답을 포함해 기본 3회까지 교정 요청한다.
- 문서 계약을 통과한 뒤 품질 검토는 기본 최대 2회 수행하며 필수 의뢰 조건이 남을 때만 두 번째 검토를 호출한다.
- 일시적인 HTTP 429는 서버가 전달한 대기시간과 지수 백오프 중 큰 값을 적용한다. 모델별 일일 요청 한도 초과는 즉시 재시도하지 않고 초안과 품질 진단을 실행 폴더에 보존한다.
- Gemini Auto 모드는 모델별 일일 한도는 즉시 다음 안정 텍스트 모델로 전환한다. RPM과 TPM 제한은 서버 대기시간과 재시도를 먼저 적용하고 계속 실패하면 다음 모델로 전환한다.
- provider에는 같은 system 지시와 user 작업을 역할별로 분리해 전달한다.
- Gemini의 STOP과 Qwen의 stop만 정상 종료로 인정하고 출력 잘림과 안전 차단을 오류로 처리한다.
- provider가 제공하는 입력, 출력, 전체 토큰과 종료 사유를 실행 로그에 기록한다.

## Excel 생성

- 원본 템플릿을 덮어쓰지 않는다.
- 프로그램 정보의 4행을 변경 파일 수만큼 복제한다.
- 셀에 `{{placeholder}}`가 있으면 이를 우선 치환하고, 현재 빈 템플릿은 기존 좌표 규약으로 채운다.
- 고정 문구와 style index를 유지하고 동적 셀만 교체한다.
- 테스트케이스와 테스트 결과의 본문 셀에는 줄바꿈을 적용한다.
- Excel 전용 XML namespace와 `mc:Ignorable` 선언을 보존하고 임시 파일의 ZIP과 시트 구조를 먼저 검증한 뒤 최종 경로로 원자적으로 이동한다.
- 최종 파일은 자동 저장하지 않고 사용자가 다운로드 버튼에서 경로와 이름을 정한다.
