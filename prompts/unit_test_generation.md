# 프롬프트 계약

{{PROMPT_METADATA}}

# 목표

개발 의뢰와 변경 근거를 함께 분석해 하나의 통합 단위테스트 문안을 작성한다. 항목 수는 커밋과 파일 수가 아니라 사용자가 수행할 독립 동작과 실패 위험으로 결정한다. 단순 변경은 억지로 늘리지 않고, 변경 파일이 많아도 개발 의뢰의 중심 업무와 핵심 위험을 최대 다섯 단계 안에 통합한다.

# 분석 순서

1. 개발 의뢰에서 시스템명과 업무 배경을 찾되, 넓은 의뢰 범위 자체를 모두 이번 테스트 대상으로 간주하지 않는다.
2. 사용자 입력 변경 요약과 선택된 Git 커밋 및 diff를 이번 변경의 우선 근거로 사용하고, 변경 목록의 신규, 변경, 삭제와 실제로 바뀐 조건, 계산, 조회, 저장, 표시 동작을 찾는다.
3. 언어와 프레임워크에 맞는 정확한 import, include, 호출 대상, endpoint, 데이터 객체와 계약으로 연결된 소스를 사용해 입력에서 결과까지의 흐름을 교차 확인한다.
4. 변경을 업무 동작별로 묶고 각 묶음에서 진입점, 분기 조건, 호출 계층, 데이터 계약, 관찰 가능한 결과를 근거 번호와 함께 내부적으로 정리한다.
5. 변경된 조건문과 예외 처리, 변경 없음 처리, 입력 경계, 권한 거부, 삭제 영향, 이름변경 호환성, 기존 기능 회귀 중 실제 근거가 있는 후보를 찾는다.
6. 정상 흐름만 반복하지 말고 변경으로 새로 생기거나 달라진 분기와 실패 가능성을 우선해 테스트 가치가 높은 대표 시나리오를 고른다.
7. 같은 동작의 화면과 서버 파일뿐 아니라 Python 모듈, SQL 객체, JSP include, SAP 프로그램과 CDS처럼 서로 연결된 구현은 별도 시나리오로 중복 계산하지 않고 실행 진입점에서 업무 및 데이터 결과까지 하나의 흐름으로 묶는다.
8. 선택하지 못한 중요한 변경 주제는 처리내역과 결과 확인에 반영하고 아무 근거 없이 정상 동작으로 뭉뚱그리지 않는다.
9. 각 문장이 근거에서 확인되는지 점검하고 확인되지 않는 고유명사, 역할, 업무 규칙, 식별자를 제거한다.
10. 변경 규모와 테스트 단계를 일대일로 대응시키지 않는다. 같은 사용자 흐름의 여러 파일은 한 단계로 통합하고, 한 파일 안에서도 서로 다른 저장, 조회, 삭제, 다운로드 분기가 있으면 필요한 만큼 분리한다.
11. 개발 의뢰의 권한, 메뉴, DB 같은 넓은 표현은 선택된 변경 파일이나 diff에서 직접 뒷받침될 때만 필수 테스트 시나리오로 채택한다.

# 필드 작성 기준

documentTitle

- 저장 파일명에 사용할 짧은 시스템명만 작성한다.
- 확장자, 단위테스트, 날짜, 시간, 괄호를 제외한 시스템명만 사용한다.
- 시스템명을 확인할 수 없으면 빈 문자열을 사용한다.

testCase.name

- 핵심 변경 목적을 드러내고 단위테스트로 끝나는 한 문구를 작성한다.
- 접수번호는 개발 의뢰에서 정확히 확인된 경우에만 포함한다.

testCase.procedure

- 한 개 이상 다섯 개 이하로 작성하고 누른다, 선택한다, 실행한다처럼 다로 끝나는 자연스러운 완결 문장으로 쓴다.
- 단순 문구 수정이나 단일 조건 변경은 한두 단계로 충분하면 늘리지 않는다. 준비, 실행, 관찰을 분리해야 실제 테스트가 쉬울 때만 단계를 추가한다.
- 여러 변경 흐름은 개발 의뢰와 직접 연결되는 핵심 사용자 동작과 위험을 최대 다섯 단계로 통합한다.
- 저장은 입력값 또는 변경 상태를 만들고 저장을 실행하도록, 조회는 조회조건을 지정하고 조회하도록, 다운로드는 조건을 지정해 파일을 내려받도록 구체적으로 쓴다.
- 삭제, 권한 거부, 입력 오류 근거가 있으면 대상과 조건을 제시하고 해당 동작을 실제로 수행하도록 쓴다.
- 변경 사항 확인, 기능 확인, 정상 동작 확인처럼 조작 대상과 관찰 결과가 없는 절차를 쓰지 않는다.

testCase.targetIds와 testCase.targetNames

- 화면, 배치 Step, 보고서 식별자와 이름이 근거에 그대로 있는 경우에만 작성한다.
- 클래스명과 파일명을 화면 이름으로 바꾸지 않는다. 불확실하면 빈 배열을 사용한다.

testCase.preconditions

- 실제 실행에 꼭 필요한 조건만 최대 다섯 개까지 작성하고 있어야 한다, 준비되어 있다처럼 다로 끝나는 자연스러운 완결 문장으로 쓴다.
- 단순 변경에 필요 없는 일반 조건을 채워 넣지 않고 권한, 비교 기준 데이터, 실행 환경처럼 절차 수행을 좌우하는 조건만 남긴다.
- 권한, 기준 데이터, 실행 환경처럼 실제 절차에 필요한 조건을 쓰되 확인되지 않은 계정명과 데이터값을 만들지 않는다.
- 클래스, 객체, 메서드, 쿼리, 파일이 구현되거나 작성되어 있어야 한다는 식의 구현 존재 여부는 테스트 사전조건으로 쓰지 않는다.
- 별도 준비 조건이 없으면 빈 배열을 사용한다.

testCase.testData

- 변경 전후 또는 조건별 결과를 구분할 수 있는 데이터를 한 문장으로 작성한다.
- 소스에 없는 실제 고객 정보, 코드값, 금액, 날짜를 만들지 않는다.
- 서로 다른 분기를 검증하는 경우 변경 있음과 변경 없음처럼 입력 조건과 기대 차이를 구분해 쓴다.
- 식별 가능한 값이 여러 개면 각 값의 의미가 드러나도록 한국어 조사와 쉼표를 사용해 자연스럽게 연결한다.
- 여러 키와 값을 조사 없이 공백으로만 이어 붙인 나열문은 쓰지 않는다.
- 별도 입력 데이터가 필요 없는 변경이면 빈 문자열을 사용한다.

testCase.expectedResult

- 한 개 또는 서로 직접 연결된 두 개의 핵심 관찰 결과를 한 문장으로 통합하고 자연스러운 한국어 서술형인 다로 끝낸다.
- 두 결과가 필요하면 쉼표와 접속 표현으로 자연스럽게 연결하되 두 줄 분량을 넘기지 않는다.
- 정상 동작한다처럼 대상이 없는 표현만 사용하지 않는다.

testCase.notes

- 꼭 필요한 미확인 사항이나 테스트 범위 제한만 한 문장으로 작성한다.
- 별도 참고사항이 없으면 빈 문자열을 사용한다.

testResult.processingDetails

- 파일별 목록이 아니라 변경 주제별로 한 개 이상 다섯 개 이하를 작성한다.
- 단순 변경은 한 항목으로 충분하면 늘리지 않고 대규모 변경은 같은 업무 흐름의 수정 사항을 합쳐 다섯 항목 안에 정리한다.
- title은 변경 대상을 짧게 쓰고 detail은 무엇이 어떻게 바뀌었는지 한 줄로 쓴다.

testResult.testDetails

- procedure와 같은 항목 수인 한 개 이상 다섯 개 이하로 작성하고 같은 순서로 대응시킨다.
- 각 항목은 관찰할 결과를 포함하며 다로 끝나는 자연스러운 완결 문장으로 쓴다.
- 무엇이 바뀌었는지 확인한다처럼 쓰지 않고 저장된 값, 조회 목록, 다운로드 파일, 삭제 후 상태, 권한 오류처럼 사용자가 판정할 대상을 쓴다.

testResult.resultChecks

- 핵심 판정 기준을 한 개 이상 다섯 개 이하로 작성한다.
- 같은 의미를 반복하지 않고 명사형 확인 문구로 간결하게 쓴다.

# 품질 점검

JSON을 출력하기 전에 다음 조건을 모두 확인한다.

1. 고유명사와 식별자는 개발 의뢰 또는 소스 근거에서 확인된다.
2. 절차와 테스트내역은 각각 한 개 이상 다섯 개 이하이고 가능한 한 같은 수와 순서로 대응하며 다른 검증 의도를 가진다.
3. 테스트 데이터와 예상결과가 변경 조건에 직접 연결된다.
4. 처리내역은 파일 수와 관계없이 업무 변경 주제로 묶였다.
5. 각 문자열 값에 콜론, 세미콜론, 가운데점, 원형 기호, 글머리 기호, 따옴표, 역따옴표, 줄바꿈, 끝 문장 부호가 없다. 문장 안의 쉼표는 자연스러운 값 구분에 사용할 수 있다.
6. 배열 문자열 앞에 번호가 없으며 설명이나 Markdown 코드 펜스가 JSON 밖에 없다.
7. 사전조건은 테스트 실행 전에 준비할 수 있는 상태이며 구현 파일이나 객체의 존재를 조건으로 쓰지 않았다.
8. 변경된 분기, 오류, 삭제, 이름변경 근거가 있는데도 정상 흐름만 반복하지 않았다.
9. 절차는 실제 조작과 입력 조건이 드러나고 테스트내역은 관찰 결과가 드러나며 서로 의미상 중복되지 않는다.
10. 처리내역과 테스트내역이 개발 의뢰의 핵심 변경 주제를 빠뜨리지 않는다.
11. 단순 변경을 억지로 여러 항목으로 늘리지 않았고 대규모 변경도 파일별 나열 없이 핵심 사용자 흐름과 위험으로 묶었다.
12. 테스트 데이터가 여러 키워드와 값을 포함할 때 조사와 쉼표로 자연스럽게 구분했다.
