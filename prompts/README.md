# 프롬프트 관리 기준

`manifest.json`이 현재 프롬프트 묶음과 컨텍스트 선별 정책, 응답 스키마의 버전을 관리한다. 실행 프롬프트에는 이 버전이 기록되므로 생성 결과의 기준을 나중에 재현할 수 있다.

## 변경 주기

- 분기마다 대표 변경 사례를 다시 검토한다.
- Excel 필드나 JSON 구조가 바뀌면 즉시 검토한다.
- scanner의 연관 파일 점수와 컨텍스트 예산이 바뀌면 대규모 변경 회귀 테스트를 실행한다.
- 사용자 문안 예시가 추가되면 자연스러운 한국어, 금지 문자, 근거 없는 식별자 검사를 보강한다.

## 버전 규칙

- JSON 필드나 의미가 호환되지 않게 바뀌면 주 버전을 올린다.
- 작성 규칙이나 선별 알고리즘이 확장되면 부 버전을 올린다.
- 오탈자처럼 결과 의미를 바꾸지 않는 수정은 패치 버전을 올린다.

## 배포 전 점검

1. 작은 단일 변경에서 import와 mapper 계약 파일이 선택되는지 확인한다.
2. 여러 변경 파일에서 각 변경 근거가 예산 안에 공정하게 포함되는지 확인한다.
3. 프론트엔드와 백엔드 파일이 많은 사례에서 파일 나열 대신 개발 의뢰의 업무 주제가 프롬프트 앞부분에 유지되는지 확인한다.
4. 응답 값에 가운데점, 원형 기호, 글머리 기호, 따옴표와 줄바꿈이 없는지 확인한다.
5. 기존 응답에서 선택 필드인 documentTitle이 없어도 파싱되는지 확인한다.
6. 파일 경로가 아닌 feat와 refactor 요약이 별도 변경 의도로 유지되는지 확인한다.
7. 변경 없음, 오류, 경계, 권한, 삭제 또는 회귀 신호가 있는 사례에서 정상 흐름만 반복하지 않는지 확인한다.
8. 구현 파일과 객체의 존재 여부를 테스트 사전조건으로 쓰지 않는지 확인한다.
9. Gemini와 Qwen이 같은 system 지시, user 작업, JSON 계약과 품질 기준을 받는지 확인한다.
10. 출력 잘림과 안전 차단이 문서 생성으로 이어지지 않는지 확인한다.
11. 저장, 조회조건, 다운로드, 삭제, 활성·비활성처럼 사용자가 명시한 각 조건에 실행 절차와 판정 기준이 모두 있는지 확인한다.
12. 파일 경로만 적힌 삭제 행을 별도 필수 테스트 흐름으로 부풀리지 않는지 확인한다.
13. 첫 품질 검토가 필수 조건을 고치지 못하면 두 번째 검토 후 문서 생성을 차단하는지 확인한다.
