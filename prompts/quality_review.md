# 단위테스트 문안 품질 검토

아래 초안은 JSON 구조 검증을 통과했다. 동일한 소스 근거와 출력 JSON 계약을 유지하면서 테스트 가치와 문장 품질을 한 번 더 검토한다.

검토 순서

1. 개발 의뢰의 핵심 변경 주제와 변경된 분기 조건을 초안이 실제로 다루는지 확인한다.
2. 정상 흐름만 반복했다면 근거가 있는 변경 없음, 오류, 경계, 권한, 삭제 영향, 이름변경 호환성, 회귀 후보를 우선 반영한다.
3. 절차와 테스트내역이 각각 한 개 이상 다섯 개 이하이고 같은 수와 순서로 대응하는지 확인한다.
4. 사전조건에서 클래스, 객체, 메서드, 쿼리, 파일의 구현 존재 여부를 제거하고 테스트 실행 전에 준비할 수 있는 상태로 바꾼다.
5. 근거에 없는 한국어 업무 사실, 역할, 계정, 외부 시스템, 상태값, 화면, API, 데이터값을 만들지 않는다.
6. 오탈자, 번역투, 모호한 정상 동작 표현을 없애고 사람이 작성한 짧고 자연스러운 한국어로 다듬는다.
7. 단순 변경을 억지로 여러 항목으로 부풀리지 않고 큰 변경도 파일 목록이 아니라 핵심 사용자 흐름과 위험을 최대 다섯 항목으로 통합한다.
8. 절차에 저장, 조회조건 지정, 다운로드, 삭제, 권한 거부, 오류 입력 등 근거가 있는 실제 조작이 나타나고 테스트내역에는 사용자가 관찰할 판정 기준이 나타나는지 확인한다.
9. 테스트 데이터의 식별 가능한 값을 한국어 조사와 쉼표로 자연스럽게 연결하고 키워드와 값을 공백으로만 이어 붙인 나열문을 고친다.
10. 예상결과는 한 개 또는 서로 직접 연결된 두 개 결과만 남겨 두 줄 분량 안으로 다듬는다.
11. 기존 출력 JSON 필드를 유지하고 새 항목 수 계약을 지키며 JSON 객체 하나만 출력한다.
12. 로컬 품질 진단에서 사용자 의뢰 기반 시나리오가 검토 필요로 표시되면 해당 조건의 실제 조작과 관찰 결과를 반드시 추가한다. 근거가 있는 조건을 누락한 채 초안과 같은 응답을 다시 내보내지 않는다.
