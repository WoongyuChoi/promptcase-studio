# 작성 목표

팀원에게 공유할 릴리즈 안내 메일을 작성한다.

- subject는 `[공유] 시스템명 핵심 변경 주제` 형식으로 60자 이내에서 작성한다.
- 시스템명과 기능명은 반드시 아래 입력 근거에서 확인되는 표현만 사용한다.
- subject에 릴리즈, 릴리즈 안내, 반영 및 확인 요청, 변경 반영 및 확인 요청 같은 상투적인 꼬리말을 넣지 않는다.
- 제목에는 프로젝트 또는 기능과 핵심 변경 주제만 간결하게 남긴다.
- body는 반드시 안녕하세요로 시작하고 감사합니다로 끝낸다.
- 첫 문단은 입력 근거의 시스템명 또는 기능명과 핵심 주제를 담은 70자 이내의 한 문장으로 쓴다.
- 첫 문단에 구축 사전 준비 및 기반 사항 반영을 위해 같은 긴 배경을 반복하거나 릴리즈했습니다라고 쓰지 않는다.
- `[변경 사항]`에는 실제 변경 주제를 최대 다섯 개까지 짧은 명사형 목록으로 정리한다.
- 변경 사항은 한 줄 90자 이내로 쓰고, 긴 원인과 구현 설명을 한 항목에 이어 붙이지 않는다.
- `[적용 범위]`에는 영향을 받는 화면, 기능 또는 업무 영역을 최대 세 개까지 짧게 적는다.
- `[확인 요청 사항]`에는 최종 단위테스트 문안에서 실제로 검증할 수 있는 내용만 두 개 이상 여덟 개 이하로 안내한다.
- 확인 요청은 일반 사용자가 그대로 따라 할 수 있는 쉬운 문장으로 작성한다.
- 데이터 입력, 행 추가, 저장, 재조회, 삭제, 결과 확인처럼 서로 다른 행동은 한 문장에 합치지 않고 각각 별도 목록으로 나눈다.
- 기술적인 계산 과정 대신 사용자가 화면에서 입력할 내용과 확인할 결과를 설명한다.
- 본문의 모든 목록은 `- `로 시작하고 한 줄은 100자를 넘기지 않는다.
- 감사합니다 바로 앞에는 `확인 중 문제나 예상과 다른 결과가 있으면 메일 또는 메신저로 알려주세요.`와 같은 연락 안내를 넣는다.
- 단순한 변경은 짧게 쓰고 파일 수가 많다는 이유로 내용을 늘리지 않는다.
- AI가 작성한 보고서처럼 설명하지 말고 같은 팀 동료에게 보내는 업무 메일 문체를 사용한다.

# 출력 전 형식 확인

- 첫 문자는 `{`, 마지막 문자는 `}`여야 한다.
- 최상위 JSON 객체를 정확히 하나만 출력하고 마지막 `}` 뒤에는 어떤 문자도 붙이지 않는다.
- subject와 body 문자열의 따옴표 및 최상위 객체의 닫는 `}`가 빠지지 않았는지 확인한다.
